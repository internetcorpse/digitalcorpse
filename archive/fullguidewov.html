Tricking Users with YouTube's Redirect System

Background

YouTube, like many large platforms, wraps outbound links in its own redirect system. The intent is benign: track click analytics, warn users when leaving the platform, and maintain referrer hygiene. But the implementation has a flaw that turns this trusted infrastructure into an open redirect vector.

I stumbled on this while analyzing phishing campaigns in the wild. I kept seeing URLs that looked like YouTube links at first glance but were routing users somewhere completely different. That got me curious about what exactly YouTube validates — or doesn't.

- - - - -

How It Works

The Redirect Endpoint

YouTube exposes a redirect endpoint at:

url 
https://www.youtube.com/redirect?q=https://evil.example.com

When a user clicks a link like this, their browser shows youtube.com momentarily — enough for an attacker to build a convincing pre-click preview. More critically, link previews in messaging apps and email clients all display the YouTube domain.

The Validation Gap

YouTube does show a warning interstitial for completely external URLs. But the validation logic has edge cases that can be bypassed — specifically around URL structures that embed YouTube subdomains or use URL encoding tricks.

bypass examples 
# basic — shows warning (expected)
https://youtube.com/redirect?q=https://evil.com

# subdomain in path — bypass in some clients
https://youtube.com/redirect?q=https://evil.com/youtube.com

# URL-encoded @ trick
https://youtube.com/redirect?q=https://youtube.com%40evil.com

# fragment abuse
https://youtube.com/redirect?q=https://evil.com%23youtube.com

The %40 trick is particularly effective. Decoded, youtube.com%40evil.com becomes youtube.com@evil.com — in URL syntax the username is youtube.com and the actual host is evil.com. Many parsers and human eyes misread this.

⚠ note These bypass techniques vary in effectiveness across browsers and client versions. The core open redirect itself remains consistent.

Proof of Concept

Minimal Python script to generate redirect URLs and test whether the interstitial is served or bypassed:

python 
import requests
from urllib.parse import quote

BASE = "https://www.youtube.com/redirect?q="

targets = [
    "https://evil.example.com",
    "https://evil.example.com/youtube.com",
    "https://youtube.com" + quote("@") + "evil.example.com",
]

for url in targets:
    full = BASE + quote(url, safe=":/")
    r = requests.get(full, allow_redirects=False)
    # 302 = direct redirect (no interstitial)
    # 200 = interstitial page served
    print(f"{r.status_code} — {full[:80]}")
"a 302 where you expected a 200 is a finding."

Impact & Scope

The impact here is primarily phishing and social engineering. An attacker can craft a URL that appears as a YouTube link in messaging app previews, passes naive URL blocklists checking only the root domain, and survives some email security scanners that whitelist known platforms.

This isn't remote code execution. It doesn't compromise YouTube accounts. But in a spear-phishing context, a believable URL is often all you need.

- - - - -

Mitigation

From a user perspective: always check the final destination URL after any redirect resolves. Browser extensions like ClearURLs can strip tracking redirects before they execute.

From a platform perspective, the fix is straightforward — strict allowlist validation of the q parameter against known YouTube-owned domains, with proper URL normalization applied before comparison.

for developers Normalize the URL first (decode, parse, re-encode), then validate against a strict allowlist. Never validate against the raw string — that's where bypasses live.
